TypeScript ⟷ Python — syntax dictionary

Side-by-side basic syntax, for Python readers learning to read TypeScript

Variables & basic types

TypeScriptPythonNotes
let x: number = 5; x: int = 5 TS needs let/const; Python doesn't
const x = 5; x = 5 # by convention only Python has no true constants — const just blocks reassignment in TS
let name: string = "Al"; name: str = "Al" Type hints look almost identical
let ok: boolean = true; ok: bool = True Capitalization differs: true/false vs True/False
let n: any = null; n = None TS also has undefined, which Python has no equivalent for
let x: number | null; x: int | None Union types use the same | syntax in modern versions of both
const x: unknown = ...; x: object = ... Loosely analogous "could be anything" types

TypeScript requires a declaration keyword (let, const, or var) every time you introduce a variable. Python just assigns. Every statement in TS also ends in a semicolon by convention — Python uses newlines and indentation instead of ; and {}.

Strings, numbers & templating

TypeScriptPythonNotes
`Hello, ${name}!` f"Hello, {name}!" Template literals (backticks) ≈ Python f-strings
"a" + "b" "a" + "b" String concatenation is the same
str.length len(s) Property vs. built-in function
str.toUpperCase() s.upper() Both are methods on the string
str.slice(1, 3) s[1:3] TS uses a method; Python uses slice syntax
Number("42") int("42") TS's Number covers ints and floats; Python splits int/float
10 % 3 10 % 3 Modulo is the same operator
10 ** 2 10 ** 2 Exponentiation is the same operator

Arrays / lists

TypeScriptPythonNotes
let arr: number[] = [1, 2, 3]; arr: list[int] = [1, 2, 3] Both are ordered, mutable, zero-indexed
Array<number> list[int] Generic-style alternate syntax for the same type
arr.push(4) arr.append(4) Add to the end
arr.pop() arr.pop() Remove and return the last item — identical
arr.length len(arr) Property vs. built-in function
arr.includes(3) 3 in arr Membership check
arr.map(x => x * 2) [x * 2 for x in arr] TS chains methods; Python prefers comprehensions
arr.filter(x => x > 0) [x for x in arr if x > 0] Same idea, different shape
arr.reduce((a, b) => a + b, 0) sum(arr) / functools.reduce(...) Python has dedicated built-ins for common cases
arr.slice(1, 3) arr[1:3] Method call vs. slice syntax
[...arr1, ...arr2] arr1 + arr2 Spread operator vs. list concatenation
arr.forEach(x => console.log(x)) for x in arr: print(x) Method callback vs. native for-loop

Objects / dictionaries

TypeScriptPythonNotes
let o = { a: 1, b: 2 }; o = {"a": 1, "b": 2} TS object keys don't need quotes; Python dict keys usually do
o.a or o["a"] o["a"] TS allows dot access on objects; Python dicts require brackets
Object.keys(o) o.keys() Static function vs. instance method
Object.values(o) o.values() Same relationship as above
Object.entries(o) o.items() Both return key/value pairs
"a" in o "a" in o Key-existence check is the same operator
delete o.a; del o["a"] Remove a key
{ ...o, c: 3 } {**o, "c": 3} Spread/merge syntax is nearly identical
Record<string, number> dict[str, int] Generic mapping type declarations
Map<string, number> dict[str, int] TS's Map class behaves like a Python dict; plain objects can too

Functions

TypeScriptPythonNotes
 
function add(a: number, b: number): number {
  return a + b;
}
 
def add(a: int, b: int) -> int:
    return a + b
 Return type goes after ) in both, with different punctuation
const add = (a: number, b: number) => a + b; add = lambda a, b: a + b Arrow function ≈ lambda, but arrows can hold full bodies too
function greet(name: string = "World") {} def greet(name: str = "World"): ... Default parameter syntax looks almost identical
function sum(...nums: number[]) {} def total(*nums): ... Rest parameters ≈ Python's *args
name?: string (optional param) name: str | None = None TS marks optionality with ?; Python uses a default + union type
function f(): void {} def f() -> None: ... "Returns nothing" annotation

TS function bodies are wrapped in { }; Python relies entirely on indentation, with no closing brace or keyword needed.

Control flow

if (x > 0) {
  ...
} else if (x < 0) {
  ...
} else {
  ...
}
 
if x > 0:
    ...
elif x < 0:
    ...
else:
    ...
 else if vs. elif; braces vs. colons + indentation 
for (let i = 0; i < 5; i++) {
  ...
}
 
for i in range(5):
    ...
 C-style for-loop vs. range() 
for (const x of arr) {
  ...
}
 
for x in arr:
    ...
 for...of iterates values, directly analogous to Python's for-loop 
for (const k in obj) {
  ...
}
 
for k in obj:
    ...
 for...in iterates keys — same as iterating a Python dict directly 
while (x < 10) {
  ...
}
 
while x < 10:
    ...
 Same concept, braces vs. colon
x === 5 x == 5 TS distinguishes == (loose) from === (strict); Python only has one equality operator
x && y / x || y / !x x and y / x or y / not x Symbolic operators vs. English keywords
cond ? a : b a if cond else b Ternary expression, reordered 
switch (x) {
  case 1:
    ...
    break;
  default:
    ...
}
 
match x:
    case 1:
        ...
    case _:
        ...
 Python 3.10+ match is the closest analogue; no fall-through by default

Classes

TypeScriptPythonNotes
 
class Dog {
  name: string;

  constructor(name: string) {
    this.name = name;
  }

  bark(): string {
    return `${this.name} says woof`;
  }
}
 
class Dog:
    def __init__(self, name: str):
        self.name = name

    def bark(self) -> str:
        return f"{self.name} says woof"
 constructor ≈ __init__; this ≈ self (explicit first param in Python)
new Dog("Rex") Dog("Rex") TS requires new; Python does not
class Puppy extends Dog {} class Puppy(Dog): ... Inheritance syntax
super(name); super().__init__(name) Calling the parent constructor
private name: string; self._name (leading underscore, by convention) TS enforces privacy at compile time; Python only signals intent
static count = 0; 
class Dog:
    count = 0
 Class-level (shared) attributes
abstract class Shape {} class Shape(ABC): ...
(from abc import ABC)		 Abstract base classes need an import in Python
interface Point { x: number; y: number; } 
class Point(Protocol):
    x: int
    y: int
or a @dataclass		 Python has no direct interface keyword — closest are Protocol or dataclasses

Types & type aliases

TypeScriptPythonNotes
type ID = string | number; ID = str | int Type alias declaration
enum Color { Red, Green, Blue } 
from enum import Enum
class Color(Enum):
    RED = 1
    GREEN = 2
    BLUE = 3
 Python's Enum needs an explicit import and value
function f<T>(x: T): T { return x; } 
def f[T](x: T) -> T:
    return x
(3.12+) or TypeVar (older)		 Generics — newer Python syntax closely mirrors TS
x as string cast(str, x)
(from typing import cast)		 Type assertion — a hint to the type checker, not a runtime conversion in either language
typeof x type(x) Runtime type inspection (different from static type annotations)
x instanceof Dog isinstance(x, Dog) Runtime type check
[number, string] (tuple type) tuple[int, str] Fixed-length, mixed-type sequence

Modules & imports

TypeScriptPythonNotes
import { foo } from "./bar"; from bar import foo Named import
import foo from "./bar"; import bar
(then bar.foo)		 TS default import vs. Python's module-level access
import * as bar from "./bar"; import bar "Import everything as a namespace" — Python modules work this way by default
export function foo() {} def foo(): ... Python exposes everything at module level automatically — no export keyword needed
export default foo; — (no direct equivalent) Python has no concept of a single "default" export per file

Async code

TypeScriptPythonNotes
 
async function getData(): Promise<string> {
  const res = await fetch(url);
  return res.text();
}
 
async def get_data() -> str:
    res = await fetch(url)
    return await res.text()
 async/await keywords are shared between the languages
Promise<number> Coroutine[Any, Any, int] or just documented as async TS's Promise type is more commonly spelled out than Python's coroutine typing
Promise.all([p1, p2]) asyncio.gather(p1, p2) Run multiple async operations concurrently

Naming & style conventions

TypeScriptPythonNotes
camelCase snake_case Variable and function naming convention
PascalCase PascalCase Class names use the same convention in both
UPPER_SNAKE_CASE UPPER_SNAKE_CASE Constants — shared convention
// comment # comment Single-line comments
/* block comment */ """docstring-style block""" Python has no true block comment — triple-quoted strings are used informally
2 spaces (common convention) 4 spaces (PEP 8) Indentation is cosmetic in TS, syntactically meaningful in Python

The biggest conceptual shift for a Python reader: TypeScript's type annotations are erased at compile time and never affect runtime behavior — it compiles down to plain JavaScript. Python's type hints are similarly non-enforcing at runtime by default, so in that respect the two are more alike than they might first appear.
